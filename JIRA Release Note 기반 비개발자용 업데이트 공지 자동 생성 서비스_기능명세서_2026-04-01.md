## 1. 프로젝트 개요
---
### 카테고리
> 생산성 / 업무

### 한 줄 정의
JIRA Release Note 기반 비개발자용 업데이트 공지 자동 생성 서비스

### 배경
IT 시스템 업데이트 공지 작성은 개발팀에게는 부담스러운 업무이며, 비개발팀에게는 기술적인 내용으로 인해 이해하기 어려운 경우가 많습니다. 이로 인해 정보 공유가 지연되거나 오해가 발생하여 업무 효율성이 저하되는 문제가 있습니다.

### 사용자 문제
개발팀은 기술적인 JIRA Release Note를 비개발자 관점으로 다시 작성하는 데 많은 시간을 소모합니다. 비개발팀은 기술적인 용어로 가득 찬 업데이트 공지를 이해하는 데 어려움을 겪고, 실제 업무 영향도를 파악하기 힘듭니다. 중요한 업데이트 내용이 비개발 팀에 효과적으로 전달되지 않아 고객 응대, 영업 전략, 운영 프로세스에 혼선이 발생할 수 있습니다. 수동적인 공지 작성 및 배포 과정에서 누락이나 오류가 발생할 가능성이 있습니다.

### 해결 방식
JIRA Release Note와 연결된 JIRA 티켓 내용을 분석하여 비개발자 친화적인 언어로 업데이트 공지 초안을 자동 생성합니다. 사용자가 초안을 직접 수정(삭제, 병합, 순서 변경, 카테고리 지정)하고, 필요시 스크린샷 및 Slack 링크를 추가하여 완성된 공지를 지정된 이메일로 발송 및 관리할 수 있도록 지원합니다.

### 차별점
JIRA Release Note URL 기반의 간편한 자동 생성: 다수의 JIRA Release URL을 입력하여 한 번에 통합된 공지를 생성할 수 있습니다. 비개발자 관점의 콘텐츠 자동 변환: 기술적 티켓 내용을 사용자 관점의 개선/변화 사항으로 자동으로 변환하여 가독성을 높입니다. 높은 사용자 자유도: 생성된 초안에 대해 사용자가 직접 병합, 삭제, 순서 변경, 카테고리 지정 등 세밀한 편집이 가능하여 최종 공지의 품질을 높일 수 있습니다. 스크린샷 및 Slack 링크 자동 포함: 필요한 시각 자료와 추가 논의 채널 링크를 자동으로 포함하여 정보 전달력을 강화합니다. 발송된 공지 관리 및 재발송 기능: 이메일 발송 이후에도 공지 내용을 관리하고 필요 시 재발송할 수 있어 효율적인 정보 관리가 가능합니다.

### 타겟 사용자
물류운영팀, 영업팀, 고객만족팀 등 IT 시스템의 변경사항을 이해하고 내외부에 전달해야 하는 비개발 직군의 팀원. 특히, JIRA를 통해 관리되는 IT 시스템의 업데이트 내용을 정기적으로 공유받는 사용자.

### 사용 시나리오
사용자(개발팀 또는 담당자)가 웹 애플리케이션에 접속합니다. JIRA Release Note URL을 입력 필드에 하나 이상 입력합니다. '업데이트 공지 초안 생성' 버튼을 클릭합니다. 시스템은 입력된 JIRA Release Note URL을 통해 JIRA 시스템에서 관련 티켓 정보를 가져오고, 각 티켓 내용을 분석하여 비개발자 친화적인 업데이트 노트 항목 초안을 자동 생성합니다. 필요한 경우 Slack 링크도 자동으로 포함됩니다. 생성된 초안 목록을 확인하며, 불필요한 항목은 삭제하고, 유사한 목적의 여러 티켓은 하나의 항목으로 병합합니다. 각 항목의 설명 내용을 수정하거나, 순서를 변경하고, 적절한 카테고리를 지정합니다. 필요한 경우 스크린샷을 추가합니다. 완성된 업데이트 공지를 최종 검토합니다. '이메일 발송' 옵션을 선택하고, 수신자 이메일 주소를 지정한 후 발송합니다. 발송된 업데이트 공지 메일은 목록으로 관리되며, 필요 시 해당 메일을 재선택하여 재발송할 수 있습니다.

### 제품 목표
비개발 팀원(고객만족팀, 물류운영팀, 영업팀 등)이 IT 시스템 업데이트 내용을 쉽고 정확하게 이해하여 업무에 활용할 수 있도록 돕고, 업데이트 공지 작성에 소요되는 시간과 노력을 자동화를 통해 절감합니다. 기술적인 내용을 사용자 친화적인 언어로 변환하여 소통의 효율성을 극대화합니다.

### 핵심 KPI
업데이트 공지 작성 시간 30% 단축, 월별 업데이트 공지 생성 건수, 사용자 만족도 점수 (설문조사 기반), 비개발자 팀의 업데이트 내용 이해도 증진 (관련 문의 감소율), 이메일 발송 기능 사용률

### 리스크/이슈
JIRA API 연동 문제: JIRA 버전 업데이트, API 변경 등으로 인해 데이터 연동이 불안정해질 수 있습니다. 자동 생성된 내용의 품질 문제: AI/ML 기반 텍스트 생성의 한계로 인해 부정확하거나 부자연스러운 내용이 생성될 수 있으며, 이에 대한 사용자 수정 작업이 예상보다 많아질 수 있습니다. 보안 및 권한 문제: JIRA 데이터 접근 및 이메일 발송 시 보안 취약점이 발생할 수 있습니다. 사용자 수용성 저하: 새로운 도구 학습에 대한 저항이나 기존 수동 방식에 대한 선호가 있을 수 있습니다. 시스템 과부하: JIRA API 호출 빈도 증가 및 대량의 데이터 처리 시 시스템 성능 저하가 발생할 수 있습니다.

### 사용자 역할

## 1. JIRA 연동 및 업데이트 공지 자동 생성
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 시작전

입력된 JIRA Release Note URL을 기반으로 JIRA 시스템에 접근하여 연결된 JIRA 티켓의 내용을 가져오고, 이를 비개발자 관점의 업데이트 공지 초안으로 자동 변환하여 생성합니다. JIRA 티켓에 첨부된 Slack 링크가 있을 경우 해당 링크도 설명 하단에 포함됩니다.

**수용 기준:**
```
1. ○ 사용자가 JIRA Release Note URL을 하나 이상 입력하면, 해당 Release에 포함된 모든 JIRA 티켓 내용이 자동으로 분석되어 업데이트 공지 초안이 생성되어야 합니다.
2. ○ 생성된 초안은 기술 용어 대신 사용자 관점의 개선 및 변화 사항을 명확히 설명해야 합니다.
3. ○ JIRA 티켓에 Slack 링크가 첨부되어 있다면, 생성된 공지 항목의 설명 하단에 해당 Slack 링크가 포함되어야 합니다.
4. ○ JIRA 연동 실패 시 사용자에게 명확한 오류 메시지를 제공해야 합니다.
```

## 2. 업데이트 공지 초안 편집 및 사용자 정의
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 시작전

생성된 업데이트 공지 초안을 사용자가 직접 편집할 수 있는 기능을 제공합니다. 여기에는 항목 삭제, 여러 항목 병합, 설명 내용 수정, 순서 변경, 카테고리 지정, 스크린샷 삽입 기능이 포함됩니다.

**수용 기준:**
```
1. ○ 사용자가 개별 업데이트 공지 항목을 선택하여 삭제할 수 있어야 합니다.
2. ○ 사용자가 두 개 이상의 업데이트 공지 항목을 선택하여 하나의 항목으로 병합하고, 병합된 항목의 내용을 편집할 수 있어야 합니다.
3. ○ 각 공지 항목의 설명 내용을 자유롭게 수정할 수 있는 에디터를 제공해야 합니다.
4. ○ 공지 항목의 순서를 드래그 앤 드롭 등으로 쉽게 변경할 수 있어야 합니다.
5. ○ 각 공지 항목에 스크린샷 이미지를 최대 5개까지 첨부할 수 있어야 합니다. 설명 섹션 하단에 보여줍니다. 
```

## 3. 이메일 발송 및 관리
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 시작전

최종 완성된 업데이트 공지를 지정된 수신자에게 이메일 형태로 발송하고, 발송된 이메일 목록을 관리하며 재발송할 수 있는 기능을 제공합니다.

**수용 기준:**
```
1. ○ 사용자가 완성된 업데이트 공지를 이메일 본문 형식으로 지정된 수신자에게 발송할 수 있어야 합니다.
2. ○ 발송된 이메일은 서비스 내에 목록 형태로 저장 및 관리되어야 합니다.
3. ○ 발송된 이메일 목록에서 특정 이메일을 선택하여 다시 발송할 수 있어야 합니다.
4. ○ 이메일 발송 시 제목, 발신자 이름, 수신자 주소 등을 설정할 수 있어야 합니다.
5. ○ 이메일 발송 성공 및 실패 여부를 사용자에게 알려야 합니다.
```

## 5. JIRA URL 입력 및 관리
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 시작전

JIRA Release Note URL을 입력하고 관리할 수 있는 기능을 제공합니다. 기본적으로 2개의 URL 입력 필드를 제공하며, 필요에 따라 추가할 수 있습니다.

**수용 기준:**
```
1. ○ 사용자가 JIRA Release Note URL을 텍스트 필드에 입력할 수 있어야 합니다.
2. ○ 기본적으로 2개의 URL 입력 필드가 제공되며, 'URL 추가' 버튼 등을 통해 필드를 무제한으로 추가할 수 있어야 합니다.
3. ○ 입력된 URL의 유효성을 기본적인 형태로 검증해야 합니다 (예: URL 형식).
```
