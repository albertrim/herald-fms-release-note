## Clarifications

### Session 2026-04-01

- Q: 공지 초안의 임시 저장 여부 → A: 단일 세션 완료 — 초안 생성~발송을 한 번에 진행. 브라우저를 닫으면 초안이 사라짐.
- Q: AI 변환 실패 시 처리 방식 → A: 자동 재시도 1회 후 실패 시 원본 JIRA 텍스트로 폴백하여 수동 편집 유도.
- Q: 이메일 발송 형식 → A: HTML 이메일 — 서식, 이미지 인라인 포함, 카테고리별 시각적 구분.
- Q: 수신자 관리 방식 → A: 최근 수신자 자동 완성 — 이전 발송에 사용한 수신자를 자동 완성으로 제안.
- Q: 공지 초안의 소유권/공유 범위 → A: 발송 이력은 모든 인증된 사용자가 조회 가능, 편집 중 초안은 생성자만 접근.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - JIRA 연동을 통한 업데이트 공지 초안 자동 생성 (Priority: P1)

담당자가 JIRA Release Note URL을 하나 이상 입력하고 '초안 생성' 버튼을 클릭하면, 시스템이 JIRA에서 해당 Release에 포함된 모든 티켓을 자동으로 조회·분석하여 비개발자가 이해할 수 있는 업데이트 공지 초안을 생성한다. 각 공지 항목은 기술 용어 대신 사용자 관점의 개선/변화 사항으로 표현되며, JIRA 티켓에 Slack 링크가 포함되어 있으면 해당 링크도 항목 하단에 자동 포함된다.

**Why this priority**: 이 서비스의 핵심 가치. URL 입력만으로 비개발자 친화적 공지 초안을 자동 생성하는 것이 전체 서비스의 존재 이유이며, 이 기능 없이는 나머지 기능이 무의미하다.

**Independent Test**: JIRA Release Note URL을 입력하고 초안 생성 버튼을 클릭하면, 비개발자 관점의 공지 항목 목록이 화면에 표시되는 것을 확인하여 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사용자가 URL 입력 페이지에 있고 기본 2개의 입력 필드가 표시된 상태, **When** 유효한 JIRA Release Note URL을 1개 입력하고 '초안 생성' 버튼을 클릭하면, **Then** 해당 Release에 포함된 모든 티켓이 분석되어 비개발자 관점의 공지 항목 초안 목록이 생성된다.
2. **Given** 사용자가 2개 이상의 JIRA Release Note URL을 입력한 상태, **When** '초안 생성' 버튼을 클릭하면, **Then** 모든 Release의 티켓이 통합 분석되어 하나의 공지 초안 목록이 생성된다.
3. **Given** JIRA 티켓에 Slack 링크가 첨부되어 있는 경우, **When** 초안이 생성되면, **Then** 해당 공지 항목의 설명 하단에 Slack 링크가 자동으로 포함된다.
4. **Given** 사용자가 JIRA Release Note URL을 입력한 상태, **When** JIRA 연동에 실패하면, **Then** 사용자에게 실패 원인을 설명하는 명확한 오류 메시지가 표시되고 URL 입력 페이지로 돌아갈 수 있다.
5. **Given** AI 변환 중 일부 티켓의 변환이 실패한 경우, **When** 자동 재시도 1회 후에도 실패하면, **Then** 해당 항목은 원본 JIRA 텍스트로 표시되고 사용자가 수동으로 편집할 수 있다.
6. **Given** 사용자가 URL 입력 페이지에 있는 상태, **When** 'URL 추가' 버튼을 클릭하면, **Then** 새로운 URL 입력 필드가 추가되며, 개별 삭제도 가능하다.
7. **Given** 사용자가 잘못된 형식의 URL을 입력한 상태, **When** '초안 생성' 버튼을 클릭하면, **Then** URL 형식 오류를 안내하고 초안 생성을 진행하지 않는다.

### User Story 2 - 공지 초안 편집 및 사용자 정의 (Priority: P2)

사용자가 자동 생성된 공지 초안 목록을 검토하면서 불필요한 항목을 삭제하고, 유사한 항목을 병합하며, 각 항목의 설명을 직접 수정할 수 있다. 또한 드래그 앤 드롭으로 항목 순서를 변경하고, 각 항목에 카테고리를 지정하며, 스크린샷 이미지를 최대 5개까지 첨부할 수 있다.

**Why this priority**: 자동 생성된 초안은 AI 기반 변환의 한계로 인해 수정이 필요할 수 있다. 편집 기능 없이는 최종 공지 품질을 보장할 수 없으므로 P1 다음으로 중요하다.

**Independent Test**: 미리 생성된 공지 초안 목록에서 항목 삭제, 병합, 수정, 순서 변경, 카테고리 지정, 스크린샷 첨부를 각각 수행하고, 변경 사항이 정상 반영되는지 확인하여 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 공지 초안 목록이 표시된 상태, **When** 특정 항목의 삭제 버튼을 클릭하면, **Then** 해당 항목이 목록에서 제거된다.
2. **Given** 공지 초안 목록에서 2개 이상의 항목을 선택한 상태, **When** '병합' 버튼을 클릭하면, **Then** 선택된 항목들이 하나의 항목으로 합쳐지고 병합된 내용을 편집할 수 있는 에디터가 표시된다.
3. **Given** 특정 공지 항목의 편집 화면에 있는 상태, **When** 설명 내용을 수정하고 저장하면, **Then** 변경된 내용이 공지 목록에 반영된다.
4. **Given** 공지 초안 목록이 표시된 상태, **When** 항목을 드래그하여 다른 위치에 드롭하면, **Then** 항목의 순서가 변경된다.
5. **Given** 특정 공지 항목의 편집 화면에 있는 상태, **When** 카테고리를 선택하면, **Then** 해당 항목에 카테고리가 지정되어 표시된다.
6. **Given** 특정 공지 항목의 편집 화면에 있는 상태, **When** 스크린샷 이미지를 첨부하면, **Then** 이미지가 설명 섹션 하단에 표시되며, 항목당 최대 5개까지 첨부 가능하다.
7. **Given** 특정 공지 항목에 이미 5개의 스크린샷이 첨부된 상태, **When** 추가 이미지를 첨부하려고 하면, **Then** 최대 개수 초과 안내 메시지가 표시되고 첨부가 거부된다.

### User Story 3 - 이메일 발송 및 관리 (Priority: P3)

사용자가 편집 완료된 업데이트 공지를 이메일 형태로 지정된 수신자에게 발송한다. 발송 전 제목, 발신자 이름, 수신자 주소를 설정하고 미리보기로 최종 검토할 수 있다. 발송된 이메일은 이력 목록으로 관리되며, 필요 시 특정 이메일을 선택하여 재발송할 수 있다.

**Why this priority**: 공지 초안의 생성(P1)과 편집(P2)이 완료된 후, 실제 비개발 팀에 전달하는 마지막 단계. 발송 기능 없이는 서비스의 최종 목적(정보 전달)을 달성할 수 없다.

**Independent Test**: 편집 완료된 공지를 이메일로 발송하고, 발송 이력에서 해당 메일을 확인한 뒤 재발송까지 수행하여 전체 흐름을 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 편집 완료된 공지가 있는 상태에서 이메일 설정 페이지에 진입하면, **When** 제목, 발신자 이름, 수신자 이메일 주소를 설정하면, **Then** 설정한 정보가 HTML 형식의 발송 미리보기에 반영된다(이미지 인라인 포함, 카테고리별 시각적 구분).
2. **Given** 이메일 설정이 완료되고 미리보기를 확인한 상태, **When** '발송' 버튼을 클릭하면, **Then** 지정된 수신자에게 이메일이 발송되고 발송 성공 안내가 표시된다.
3. **Given** 이메일 발송 중 오류가 발생한 경우, **When** 발송이 실패하면, **Then** 사용자에게 실패 사유를 포함한 안내 메시지가 표시되고 이메일 설정 페이지로 돌아갈 수 있다.
4. **Given** 대시보드의 발송 이력 목록에 이전에 발송한 이메일이 있는 상태, **When** 특정 이메일의 상세 보기를 선택하면, **Then** 발송된 공지 내용, 수신자, 발송일시 등 상세 정보를 확인할 수 있다.
5. **Given** 발송 이력 상세 보기에서 특정 이메일을 확인한 상태, **When** '재발송' 버튼을 클릭하면, **Then** 해당 공지 내용으로 이메일 설정 페이지가 열려 수신자를 재설정 후 발송할 수 있다.
6. **Given** 이메일 수신자 입력 필드에 포커스한 상태, **When** 이전 발송에 사용한 수신자 이메일이 존재하면, **Then** 자동 완성 목록으로 제안되어 빠르게 선택할 수 있다.

### User Story 4 - 사용자 인증 및 권한 관리 (Priority: P4)

사용자가 이메일과 비밀번호로 회원가입 및 로그인하여 서비스에 접근한다. 관리자 역할은 사용자 관리 등 추가 기능에 접근할 수 있고, 일반 사용자 역할은 공지 생성·편집·발송 기능을 사용할 수 있다. 인증되지 않은 사용자는 어떤 기능에도 접근할 수 없다.

**Why this priority**: 보안과 접근 제어를 위한 기반 기능이지만, 핵심 비즈니스 가치(공지 생성/편집/발송)와 비교하면 우선순위가 낮다. 초기 MVP에서는 간단한 인증만으로도 서비스를 운영할 수 있다.

**Independent Test**: 회원가입 후 로그인하여 대시보드에 접근하고, 로그아웃 시 기능 접근이 차단되는 것을 확인하여 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 서비스 접속 시, **When** 인증되지 않은 사용자가 기능 페이지에 접근하려고 하면, **Then** 로그인 페이지로 이동된다.
2. **Given** 회원가입 페이지에서 이메일과 비밀번호를 입력한 상태, **When** 회원가입을 실행하면, **Then** 계정이 생성되고 로그인 페이지로 이동된다.
3. **Given** 로그인 페이지에서 올바른 인증 정보를 입력한 상태, **When** 로그인을 실행하면, **Then** 대시보드 페이지로 이동된다.
4. **Given** 잘못된 인증 정보를 입력한 상태, **When** 로그인을 실행하면, **Then** 로그인 실패 안내 메시지가 표시되고 로그인 페이지에 머문다.
5. **Given** 일반 사용자 역할로 로그인한 상태, **When** 관리자 전용 기능에 접근하려고 하면, **Then** 접근이 거부되고 권한 부족 안내가 표시된다.
6. **Given** 로그인된 상태, **When** 로그아웃을 실행하면, **Then** 세션이 종료되고 로그인 페이지로 이동된다.

### Edge Cases

- JIRA Release Note URL이 유효한 형식이지만 존재하지 않는 Release를 가리키는 경우, 시스템은 "해당 Release를 찾을 수 없습니다"라는 메시지를 표시한다.
- JIRA Release에 연결된 티켓이 0개인 경우, 시스템은 "연결된 티켓이 없습니다"라는 안내를 표시하고 빈 초안을 생성하지 않는다.
- 동일한 JIRA Release Note URL을 중복 입력한 경우, 시스템은 중복을 감지하고 안내 메시지를 표시한다.
- 초안 편집 중 모든 항목을 삭제한 경우, 이메일 발송 버튼이 비활성화되고 "발송할 항목이 없습니다"라는 안내가 표시된다.
- 이메일 수신자 주소가 잘못된 형식인 경우, 발송 실행 전에 유효성 오류를 표시한다.
- 스크린샷 첨부 시 허용되지 않는 파일 형식(예: 실행 파일)을 업로드하려는 경우, 지원 형식 안내와 함께 거부한다.
- 동시에 여러 사용자가 같은 공지를 편집하는 경우는 v1 범위에서 고려하지 않는다(단일 사용자 편집 기준).
- AI 변환 중 전체 티켓 변환이 실패하고 재시도에서도 실패한 경우, 모든 항목이 원본 JIRA 텍스트로 표시되며 사용자에게 수동 편집이 필요하다는 안내를 제공한다.
- 브라우저를 닫거나 세션이 만료되면 편집 중인 공지 초안은 소실된다(임시 저장 미지원).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 JIRA Release Note URL을 입력받아 해당 Release에 포함된 모든 JIRA 티켓을 자동으로 조회해야 한다.
- **FR-002**: 시스템은 JIRA 티켓의 기술적 내용을 비개발자가 이해할 수 있는 사용자 관점의 개선/변화 사항으로 자동 변환해야 한다.
- **FR-003**: 시스템은 JIRA 티켓에 포함된 Slack 링크를 자동으로 추출하여 공지 항목 설명 하단에 포함해야 한다.
- **FR-004**: 시스템은 JIRA 연동 실패 시 원인을 포함한 명확한 오류 메시지를 사용자에게 제공해야 한다.
- **FR-004a**: AI 변환 실패 시 자동 재시도를 1회 수행하고, 재시도 후에도 실패한 항목은 원본 JIRA 텍스트로 표시하여 수동 편집을 유도해야 한다.
- **FR-005**: 시스템은 기본 2개의 URL 입력 필드를 제공하고, 사용자가 필드를 동적으로 추가/삭제할 수 있어야 한다.
- **FR-006**: 시스템은 입력된 URL의 형식을 검증하고 잘못된 형식에 대해 안내해야 한다.
- **FR-007**: 사용자는 생성된 공지 항목을 개별 삭제할 수 있어야 한다.
- **FR-008**: 사용자는 2개 이상의 공지 항목을 선택하여 하나로 병합하고 병합된 내용을 편집할 수 있어야 한다.
- **FR-009**: 사용자는 공지 항목의 설명 내용을 자유롭게 수정할 수 있는 에디터를 사용할 수 있어야 한다.
- **FR-010**: 사용자는 드래그 앤 드롭으로 공지 항목의 순서를 변경할 수 있어야 한다.
- **FR-011**: 사용자는 각 공지 항목에 카테고리를 지정할 수 있어야 한다.
- **FR-012**: 사용자는 각 공지 항목에 스크린샷 이미지를 최대 5개까지 첨부할 수 있어야 하며, 이미지는 설명 하단에 표시된다.
- **FR-013**: 사용자는 완성된 공지를 HTML 이메일 본문 형식(이미지 인라인 포함, 카테고리별 시각적 구분)으로 지정된 수신자에게 발송할 수 있어야 한다.
- **FR-014**: 시스템은 이메일 발송 시 제목, 발신자 이름, 수신자 주소를 설정할 수 있어야 한다.
- **FR-014a**: 수신자 입력 시 이전 발송에 사용한 수신자 이메일을 자동 완성으로 제안해야 한다.
- **FR-015**: 시스템은 발송된 이메일을 목록 형태로 저장하고 관리해야 한다.
- **FR-016**: 사용자는 발송 이력에서 특정 이메일을 선택하여 재발송할 수 있어야 한다.
- **FR-017**: 시스템은 이메일 발송 성공 및 실패 여부를 사용자에게 알려야 한다.
- **FR-018**: 사용자는 이메일과 비밀번호로 회원가입 및 로그인할 수 있어야 한다.
- **FR-019**: 시스템은 관리자와 일반 사용자 두 가지 역할을 구분하고, 역할에 따라 접근 가능한 기능을 제한해야 한다.
- **FR-020**: 인증되지 않은 사용자는 서비스의 어떤 기능에도 접근할 수 없어야 한다.
- **FR-021**: 발송 이력은 모든 인증된 사용자가 조회할 수 있어야 한다. 편집 중인 공지 초안은 생성자만 접근할 수 있어야 한다.

### Key Entities

- **사용자(User)**: 서비스에 접근하는 사람. 이메일, 비밀번호, 역할(관리자/일반 사용자)을 속성으로 가진다.
- **공지 초안(Draft Notice)**: JIRA 티켓 분석을 통해 자동 생성된 업데이트 공지. 여러 개의 공지 항목으로 구성된다. 생성 일시, 원본 JIRA Release Note URL 목록, 생성자를 속성으로 가진다. 세션 기반으로 존재하며 영구 저장되지 않는다(단일 세션 완료 방식). 생성자만 접근 가능하다.
- **공지 항목(Notice Item)**: 개별 업데이트 항목. 제목, 설명, 카테고리, 순서, 원본 JIRA 티켓 참조, Slack 링크, 첨부 스크린샷 목록을 속성으로 가진다. 공지 초안에 속한다.
- **발송 이력(Send History)**: 이메일로 발송된 공지의 이력. 발송일시, 제목, 발신자 이름, 수신자 목록, 발송 상태(성공/실패), 공지 내용 스냅샷을 속성으로 가진다. 모든 인증된 사용자가 조회 가능하다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 JIRA Release Note URL 입력부터 공지 초안 확인까지 3분 이내에 완료할 수 있다.
- **SC-002**: 자동 생성된 공지 초안의 80% 이상 항목이 사용자 수정 없이도 비개발자가 이해 가능한 수준이다.
- **SC-003**: 기존 수동 작성 방식 대비 공지 작성 소요 시간이 30% 이상 단축된다.
- **SC-004**: 공지 초안 편집(삭제/병합/수정/순서변경/카테고리/스크린샷) 전 과정을 10분 이내에 완료할 수 있다.
- **SC-005**: 이메일 발송 설정부터 발송 완료까지 2분 이내에 완료할 수 있다.
- **SC-006**: 발송된 이메일의 재발송을 1분 이내에 완료할 수 있다.
- **SC-007**: 신규 사용자가 별도 교육 없이 첫 공지 생성 및 발송을 15분 이내에 완료할 수 있다.

## Assumptions

- 대상 사용자(물류운영팀, 영업팀, 고객만족팀 등)는 안정적인 인터넷 연결 환경에서 웹 브라우저를 통해 서비스를 사용한다.
- JIRA 시스템은 조직 내부에서 운영 중이며, 시스템이 JIRA API에 접근할 수 있는 인증 정보가 사전에 설정되어 있다(조직 공용 서비스 계정 방식).
- 모바일 지원은 v1 범위에 포함되지 않으며, 데스크톱 웹 브라우저만 대상으로 한다.
- 공지 항목의 카테고리는 관리자가 사전 정의한 목록에서 선택하는 방식이다 (예: 기능 개선, 버그 수정, 신규 기능, UI/UX 변경 등).
- 이메일 발송을 위한 메일 서버(SMTP 등) 설정은 시스템 관리자가 사전에 구성해 둔다.
- 동시 편집(여러 사용자가 같은 공지를 동시 수정)은 v1 범위에서 지원하지 않는다.
- 공지 초안은 단일 세션 완료 방식이며, 임시 저장 기능은 v1 범위에 포함되지 않는다.
- 스크린샷으로 첨부 가능한 이미지 형식은 일반적인 웹 이미지 형식(PNG, JPG, GIF 등)으로 제한된다.
- 관리자 역할의 추가 기능은 사용자 관리(역할 변경, 계정 비활성화)와 카테고리 관리로 한정한다.
